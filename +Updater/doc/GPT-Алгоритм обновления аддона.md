В целом алгоритм правильный. Для аддона, который уже загружен в nanoCAD, схема с `.bak` — один из самых безопасных вариантов. Я бы немного изменил порядок и добавил проверки.

Оптимальная последовательность:

1. **Скачать архив во временный каталог**
```
%TEMP%\Product_Update\version.zip
```

2. **Проверить архив**
- размер;
- контрольную сумму (желательно);
- наличие ожидаемой структуры;
- версию пакета.

3. **Распаковать во временный каталог**
Например:
```
%TEMP%\Product_Update\Extract\
    SpecSPDS\
       *.dll
    Menu\
    Icons\
    *.package
```

Не распаковывать сразу поверх рабочего каталога.

4. **Проверить, что распаковка успешна**
Например:
- все обязательные DLL присутствуют;
- версия сборки выше текущей.

5. **Сделать резервные копии старых файлов**
Только тех, которые будут заменены:

```
MyAddon.dll       → MyAddon.dll.bak
MyAddon.Core.dll  → MyAddon.Core.dll.bak
```

Лучше не переименовывать весь каталог целиком.

6. **Переместить новую структуру поверх старой**
Из временного каталога:

```
Extract\
    ↓
AddonRoot\
```

С сохранением структуры каталогов.

7. **Удалить временный каталог**

8. **При следующем запуске nanoCAD**
Перед загрузкой аддона:

- удалить все `*.bak`;
- удалить пустые каталоги снизу вверх.

Есть один важный момент: если DLL уже загружены в процесс nanoCAD, переименование обычно пройдет нормально, потому что Windows держит открытый файл через handle, но сам файл можно переименовать. Новая DLL появится рядом, а старая останется как `.bak`.

Я бы добавил ещё файл состояния:

```
update.state
```

Например:

```json
{
  "version": "26.1.5",
  "backupCreated": true,
  "completed": true
}
```

Зачем:
- если nanoCAD аварийно завершился во время обновления;
- если не удалось заменить один файл;
- можно восстановить предыдущую версию.

Ещё лучше делать так:

```
Addon\
 ├─ Current\
 │   ├─ dll
 │   └─ resources
 │
 ├─ Update\
 │   └─ new version
 │
 └─ Backup\
     └─ old version
```

Но для nanoCAD это сложнее, потому что загрузчик ожидает конкретные пути.

Для твоей схемы с `.package` и ресурсами рядом я бы оставил именно:

```
Скачать

Распаковать во временный каталог
 ↓
Проверить
 ↓
*.dll → *.dll.bak
 ↓
Копировать новую структуру поверх
 ↓
При следующем старте удалить *.bak и пустые папки
```

Это простой и достаточно надежный вариант. Я бы только не делал удаление `.bak` сразу после обновления — первый запуск новой версии лучше оставить с возможностью отката.

Да, в дистрибутиве 7-Zip действительно есть **DLL-библиотеки**, но это не готовые .NET-библиотеки.

У 7-Zip есть несколько вариантов:

1. **7z.exe / 7zz.exe**
- консольный интерфейс;
- самый простой вариант для твоего updater;
- запуск через `ProcessStartInfo`.

2. **7z.dll / 7zxa.dll**
- это нативные C/C++ библиотеки;
- через них можно работать с архивами программно;
- прямого .NET API нет.

3. **LZMA SDK**
- исходники и библиотеки для интеграции;
- есть C/C++ и C# примеры;
- это не полноценная оболочка 7z.

То есть чего-то вроде:

```csharp
using SevenZip;

SevenZip.Extract("update.7z");
```

в официальном 7-Zip нет.

Есть сторонние обертки:

- **SevenZipSharp**
- **SharpCompress**
- **7ZipExtractor**

Но у них свои нюансы.

Для твоего случая (обновление nanoCAD-аддонов) я бы не тащил нативные DLL. У тебя уже есть логика:

```
Updater.dll
    |
    +-- найти 7zz.exe
    |
    +-- распаковать временно
    |
    +-- заменить файлы
```

Это надежнее.

Можно сделать так:

```
Addon\
 ├─ Updater.dll
 ├─ Tools\
 │   ├─ x64\
 │   │   └─ 7zz.exe
 │   └─ x86\
 │       └─ 7zz.exe
```

И выбор:

```csharp
var sevenZip = Environment.Is64BitProcess
    ? "Tools\\x64\\7zz.exe"
    : "Tools\\x86\\7zz.exe";
```

Но с учетом nanoCAD 23/24/25/26 я бы ориентировался не на разрядность Windows, а на **разрядность процесса nanoCAD**, потому что твой updater будет работать внутри него.

Ещё вариант — вообще не хранить 7zz.exe в аддоне, а искать установленный:

```
C:\Program Files\7-Zip\7z.exe
```

но для корпоративных машин это менее надежно.

Для твоего updater я бы сделал:
- `7zz.exe` положить рядом с Updater.dll;
- проверять наличие;
- если нет — использовать встроенный fallback через `System.IO.Compression` для zip.

Кстати, у 7-Zip сейчас есть `7zz.exe` — это кроссплатформенная консольная версия, её как раз удобно встраивать в такие утилиты.

Да, но с уточнением: **`7zz.exe` кроссплатформенный не в смысле "один exe везде"**.

Это отдельные сборки под разные ОС:

- Windows:
  - `7zz.exe`
- Linux:
  - `7zz`
- macOS:
  - `7zz`

Исходный код один, но бинарники разные.

Для Windows-аддона nanoCAD тебе это не дает особых преимуществ — нужен именно Windows x64 вариант.

По разрядности:
- `7zz.exe` сейчас в основном выпускается как **64-bit**;
- для старых 32-битных систем есть отдельные варианты (через старые сборки 7-Zip).

В твоем случае с nanoCAD я бы сделал:

```
Updater.dll
Tools
 └── 7z
     └── win-x64
          └── 7zz.exe
```

И запуск:

```csharp
var psi = new ProcessStartInfo
{
    FileName = sevenZipPath,
    Arguments = $"x \"{archive}\" -o\"{target}\" -y",
    CreateNoWindow = true,
    UseShellExecute = false
};
```

Причем `7zz.exe` можно спокойно положить внутрь пакета аддона — лицензия 7-Zip позволяет распространение (LGPL + часть кода под BSD).

Ещё момент: у тебя updater — **.NET DLL внутри nanoCAD**, поэтому лучше использовать `7zz.exe`, а не `7z.dll`. С DLL пришлось бы:
- разбираться с P/Invoke;
- грузить native библиотеку;
- учитывать x86/x64;
- ловить конфликты загрузки.

С внешним `7zz.exe` граница проще и надежнее.

Да. `7zz.exe` — это консольная оболочка (CLI), но она **не является просто "тонкой обёрткой" над `7z.dll`**.

Есть два варианта использования движка 7-Zip:

1. **7zz.exe**
   - внутри уже содержит код для работы с архивами;
   - не требует `7z.dll`;
   - это самостоятельный исполняемый файл.

2. **7z.dll**
   - библиотека с COM-интерфейсами;
   - предназначена для интеграции в другие приложения;
   - требует писать код для работы с API.

То есть если ты запускаешь:

```cmd
7zz x update.7z
```

никакие `7z.dll` рядом не нужны.

Именно поэтому `7zz.exe` удобно распространять как единственный файл.

Для updater это обычно лучший вариант:
- один файл;
- не нужно P/Invoke;
- не нужно COM;
- поддерживает все форматы, которые умеет 7-Zip;
- обновить версию архиватора можно простой заменой `7zz.exe`.

Если же хочется полностью отказаться от запуска внешнего процесса, тогда придется использовать `7z.dll` (через собственную обертку или стороннюю библиотеку) либо выбрать другую библиотеку, например SharpCompress. Но с точки зрения простоты и надежности для обновлятора `7zz.exe` обычно выигрывает.

Да, это хорошая идея. По сути ты описал **журнал транзакции (transaction log)**. Именно так работают многие установщики и файловые системы.

### Почему `.bak` недостаточно

Если просто переименовать

```
A.dll -> A.dll.bak
```

и затем произойдет ошибка (отключение питания, исключение, нехватка места и т.д.), то при следующем запуске ты уже не знаешь:

- какие файлы были переименованы;
- какие уже заменены;
- какие нужно восстановить.

Если же есть журнал, то состояние всегда можно восстановить.

### Я бы сделал так

Перед каждым действием записывать его в журнал и сразу сбрасывать на диск (`Flush()`).

Например

```
update.rollback

R|Addon.dll|Addon.dll.bak
R|Core.dll|Core.dll.bak
R|Menu.dll|Menu.dll.bak
```

где `R` = Rename.

После успешного завершения обновления файл удаляется.

### При запуске

Если найден `update.rollback`, значит прошлое обновление завершилось аварийно.

Тогда:

```
foreach запись
    если backup существует
        если новый файл существует
            удалить новый
        вернуть backup обратно
```

После успешного отката удалить журнал.

---

### Но можно сделать еще лучше

Я бы писал не просто переименование, а операции.

Например

```
RENAME|Addon.dll|Addon.dll.bak
COPY|Temp\Addon.dll|Addon.dll
DELETE|Old.config
```

Тогда можно восстановить практически любое состояние.

### Когда откатывать

Я бы делал **сразу**, если произошла ошибка.

```
Rename старые
↓
Ошибка
↓
Откатить все rename
↓
Сообщить пользователю
```

Тогда каталог не остается в полурабочем состоянии.

При следующем запуске журнал нужен уже как **страховка** на случай:

- отключения питания;
- завершения процесса;
- BSOD;
- зависания nanoCAD;
- убийства процесса диспетчером задач.

То есть схема получается такой:

```
создать журнал
↓
rename файлов (каждый rename записывать в журнал)
↓
копирование новых файлов
↓
если ошибка
    откат по журналу
    оставить журнал, если откат не удалось завершить полностью
иначе
    удалить журнал
```

Это дает почти транзакционное обновление.

### Я бы даже немного изменил алгоритм

1. Создать `update.rollback`.
2. Записывать в него каждое успешное переименование.
3. Копировать новые файлы.
4. При любой ошибке немедленно выполнить откат по журналу.
5. Если откат завершился успешно — удалить журнал.
6. Если процесс аварийно завершился до отката, при следующем запуске обнаружить журнал, выполнить откат и только после этого разрешить новое обновление.

Такая схема достаточно простая в реализации и при этом устойчива практически ко всем реальным сбоям во время обновления.
